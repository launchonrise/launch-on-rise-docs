RISE

A fair-launch token launchpad on Solana. Built to make bundling, sniping, and whale concentration structurally impossible — enforced at the smart contract level.

SOLANA ANTI-BUNDLING OPEN SOURCE ANCHOR 0.32.1 PROGRESSIVE UNLOCK MIT LICENSE

The current Solana memecoin launchpad landscape has one dominant platform — but it has left the door wide open for bundlers, snipers, and whale wallets to extract value from regular participants. RISE is built specifically to close those doors.

Every protection in RISE is enforced at the smart contract level. Not the frontend. Not an off-chain rule. The actual on-chain program code that no user, admin, or frontend can bypass.

Core Principle

No bots. No bundlers. No whale wallets controlling 50% of the supply on day one. Every participant gets an equal shot at every launch.

01 The Problem

Four structural issues plague every existing Solana launchpad. RISE eliminates all four at the contract level — not the UI.

01
Bundling
Automated tools execute hundreds of buys in the same Solana block as a launch. A single actor acquires 40–60% of supply before any regular user can participate. The token pumps briefly as the bundler sells into demand, then collapses — leaving retail holders with near-worthless tokens.
02
Whale Concentration
No existing launchpad caps how much a single wallet can hold. When one wallet controls 20–30% of a token's supply, every other holder's outcome is entirely dependent on that wallet's decision to sell.
03
Supply Sniping
Bots monitor the Solana mempool and execute buys in the same block as token creation — acquiring tokens before the launch is visible to regular users. With 100% of supply available from block 0, there is no structural defence against this on existing platforms.
04
Spam Launches
Near-zero deploy costs flood existing platforms with thousands of worthless tokens daily. This buries genuine projects, degrades the user experience, and trains the community to expect most launches to be low-effort or fraudulent.

02 Anti-Bundling System

RISE implements a two-layer mechanism that makes large coordinated buys structurally unprofitable — enforced at the program level on every transaction without exception.

Layer 1 — Per-block buy limit

Each wallet is limited to 2 buy transactions per Solana block (~400ms). Even a bundler operating 100 wallets simultaneously can only accumulate a fraction of what they could on an unrestricted platform. The per-block limit is checked against the on-chain slot clock and cannot be circumvented.

Layer 2 — Launch cooldown

For the first 60 seconds after a token is created, each wallet is capped at a maximum purchase of 1% of total supply per transaction. The contract rejects any transaction that violates this rule — no exceptions, no admin override.

On-Chain Enforcement

Both layers are enforced by the smart contract. They cannot be bypassed by using a different frontend, calling the contract directly, or any other method.

Even if a bundler acquires every token in the launch tranche, they hold only 15% of total supply. The resulting price spike from doing so makes acquiring further tranches economically unviable — the mechanism defeats bundling at a mathematical level.

04 Wallet Hold Cap — 5%

No single wallet can hold more than 5% of a token's total supply at any time. This limit is enforced on every buy transaction at the program level. If a purchase would push a wallet above 5%, the transaction is automatically rejected by the contract.

The developer wallet that creates a token is subject to the exact same 5% cap as every other participant. There are no special permissions, no admin overrides, no exceptions for the creator. The cap is publicly verifiable on-chain by anyone.

Design Rationale

5% is large enough for a genuine early believer to take a meaningful position. Small enough that no wallet holds a controlling stake or creates single-point price dependency for the rest of the community.

06 Bonding Curve

RISE uses a constant product bonding curve — the same battle-tested mathematical model used by Uniswap and pump.fun. Price is determined entirely by supply and demand with no order books or market makers required.

virtual_sol_reserves × virtual_token_reserves = k

As users buy tokens, SOL reserves increase and token reserves decrease — price rises automatically. As users sell, the reverse occurs. The curve is fully autonomous and transparent on-chain.

08 Security

  • On-chain enforcement. Every safety rule — wallet caps, supply locks, fee routing — is enforced by the smart contract. No bypassing via direct contract calls, alternative frontends, or any other method.
  • Open source. The entire codebase is public on GitHub. Anyone can read, audit, fork, or contribute. Transparency is a security feature, not a liability.
  • Full test suite. Every instruction has corresponding tests covering normal operation, error cases, and attempted bypass of safety mechanisms. Tests run on every commit via GitHub Actions CI.
  • Devnet beta. RISE runs a full public beta on Solana devnet before any mainnet deployment. Real bugs are found and fixed before real funds are involved.
  • Limited admin scope. The RISE admin wallet can pause individual token pools in an emergency but cannot access user funds, modify token parameters post-creation, or bypass wallet caps under any circumstance.
  • Vanity program address. The RISE program address ends in the letters rise — making the contract recognizable and significantly harder to spoof with a lookalike deployment.

11 Community & Transparency

RISE is built in public and for the community. Full visibility into the codebase, treasury, and development roadmap from day one.

  • All smart contract code is public on GitHub before mainnet launch.
  • The program address (ending in "rise") is published before deployment.
  • The treasury wallet address is public. All inflows and outflows are on-chain.
  • The dev wallet for any RISE-created tokens is publicly disclosed.
  • Fee changes require a 7-day community notice period before taking effect.
  • No private pre-sales. No insider allocations. No hidden wallets.
  • Long-term governance decisions are put to community vote.
Our Commitment

Every design decision in RISE — the anti-bundling system, the progressive supply unlock, the fee structure — was made with the community's interests as the primary constraint. The best launchpad is one the community trusts completely.
